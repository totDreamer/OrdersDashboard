Описание создания и возникших в процессе трудностей:


В первую очередь, загрузил задание в GPT и попросил дать мне структуру проекта и пошаговый план на выполнение.
Создал аккаунты, бота.
Далее, подготовил все необходимые данные для интеграции сервисов между собой - url, api ключи.
Сформировал скелет проекта из папок и файлов, согласно структуре, начал заниматься бэкэнд частью.

Подготовил скрипт для загрузки заказов через API - изначально использовал библиотеку requests, но с ней ничего не вышло. Залез в документацию RetailCRM, нашел там их библиотеку для работы через API.
Полазив в исходном коде, нашел нужные мне функции, дописал скрипт с новой библиотекой - снова возникла ошибка.
Выяснил, что связана с тем, что в демо-версии RetailCRM нельзя поставить отличный от основного "orferType", а в предоставленных данных он изменен на "eshop-individual". Добавил обработку при заборе данных.
Перешел к фронтэнд части.

В Supabase сформировал проект и таблицу с необходимыми ключами.
Вновь вернулся к бэкэнд части - написал скрипт для синхронизации данных между RetailCRM и Supabase - синхронизация прошла успешно.

Создал веб-страницу с дашбордом, настроил интеграцию с БД в Supabase, привел ее в читаемое состояние с возможностью переключения отображения графика по кол-ву заказов/общей сумме за день. 
Показалось логичнее сделать так, чем пихать все данные в одно отображение.

Приступил к деплою в Vercel - произошла ошибка - нужно было убрать строгую типизацию в одном из фрагментов кода. После этого, деплой завершился  успешно.
Понял, что читаемость при выборе конкретной даты на графике оставляет желать лучшего - сделал текст более читаемым, заменил "служебные" обозначения на понятные всем.
Попробовал открыть с телефона - читаемость снова стала плохой - добавил адаптацию под мобильные устройства.

Далее, перешел к настройке уведомлений для Telegram-бота. Решил, что добавлю этот функционал в скрипт синхронизации.
В процессе тестирования выяснил, что при синхронизации данные в БД дублируются - задал правило по уникальным ключам, чтоб избежать дублей, добавил обработку исключений в скрипт.
Уведомления отсылались каждый раз при переходе порога суммы, попытался исправить, отправляя уведомление только после успешной записи в таблицу - решение не дало желаемого результата.
Изменил логику проверки - добавил в БД ключ успешной отправки со значением по умолчанию False. 
Поменял скрипт, добавив проверку на то, отправлялось ли уже уведомление и больше ли сумма заказа установленного порога.

На этом все) 
Надеюсь на дальнейшее сотрудничество!
